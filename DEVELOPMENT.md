# Development

This document covers building, testing, and releasing the Universally WordPress plugin. For end-user info, see [README.md](README.md).

## Layout

```
plugin/                  # plugin source (ships to wp.org)
  universally.php        # plugin bootstrap
  app/                   # PHP classes (PSR-4, `Universally\` namespace)
  includes/              # procedural helpers, REST entry, conflict handler
  panel/js/              # TypeScript/React sources for the admin panel
  panel/build/           # compiled JS/CSS (generated by `npm run build`)
  panel/src/             # PHP-side panel framework
  assets/                # static plugin assets (icons, switcher JS, etc.)
  languages/             # translations and POT file
  readme.txt             # wp.org listing
assets/                  # wp.org plugin page assets (banners, icons, screenshots)
scripts/release.sh       # release entry — bumps, builds, tags, pushes
docker-compose.yml       # local WordPress for quick testing
```

## Local build

Node.js 18+ and PHP 7.4+ are required.

```sh
cd plugin
npm install
npm run build       # production build
npm run start       # watch mode
composer install    # PHP autoloader
```

`npm run build` regenerates everything inside `plugin/panel/build/` (compiled JS, CSS, source maps, and `*.asset.php` dependency files) from the sources in `plugin/panel/js/` using [@wordpress/scripts](https://www.npmjs.com/package/@wordpress/scripts).

## Test in a real WordPress

```sh
docker compose up -d
open http://localhost:8765
```

Run the installer, then activate the plugin from `Plugins → Installed Plugins`. The plugin directory is bind-mounted live, so PHP changes take effect immediately; TS/SCSS changes require `npm run build` (or `npm run start` for watch mode).

```sh
docker compose down -v   # wipe DB + WP install when done
```

## Releasing — TL;DR

Two steps:

1. **Locally**, create the tag:
   ```sh
   npm run release
   ```
   Follow the prompts (bump type, channel, type-to-confirm). When done, the tag is on GitHub but **nothing has shipped yet**.

2. **On GitHub**, deploy it:
   - Open **Actions → Release → Run workflow** ([direct link](https://github.com/awesomemotive/universally-plugin/actions/workflows/release.yml)).
   - Pick the tag from the dropdown (latest 5 are listed; older ones go in `custom_tag`).
   - Run.

That's it. CI builds, runs gates, ships:
- Stable tag → wp.org SVN + GitHub Release.
- Beta/RC tag → GitHub Release only (no wp.org).

---

### Details — `npm run release`

Two-step interactive menu — first pick the **bump type**, then the **channel**:

**Bump type:**
- `1) patch` — `1.0.0 → 1.0.1`
- `2) minor` — `1.0.0 → 1.1.0`
- `3) major` — `1.0.0 → 2.0.0` (extra confirmation)
- `4) custom` — enter the exact version (skips channel prompt)
- `5) rebuild` — force re-release of the current version (skips channel prompt; replaces the tag)

**Channel** (only shown for patch / minor / major):
- `1) stable` — `vX.Y.Z`
- `2) beta` — `vX.Y.Z-beta.N` (auto-increments N from existing beta tags for that base)
- `3) rc` — `vX.Y.Z-rc.N`

Examples:
- patch + stable from `1.0.1` → `1.0.2`
- patch + beta from `1.0.1` → `1.0.2-beta.1`
- minor + rc from `1.0.1` → `1.1.0-rc.1` (or `.2` if `v1.1.0-rc.1` already exists)

A red confirmation banner appears before anything is pushed; you have to type back `release <version>` to proceed. The script bumps `plugin/universally.php`, `plugin/package.json`, and `plugin/readme.txt` (`Stable tag` only for stable releases), commits, creates an annotated `vX.Y.Z` tag, and pushes. It does **not** build and does **not** deploy.

### Details — Deploy workflow

`release.yml` is manual-only — tag pushes don't auto-ship. When you run it:

1. CI checks out the exact tag you picked.
2. `scripts/build.sh` produces the production zip (same script local `npm run build` runs).
3. phpcs (PHP 7.4 + security) and WP Plugin Check gate the release.
4. Stable tag → wp.org SVN deploy via [10up/action-wordpress-plugin-deploy](https://github.com/10up/action-wordpress-plugin-deploy). Prereleases skip this step.
5. GitHub Release published with the zip + a one-click WordPress Playground link.

The dropdown of recent tags is kept current by `.github/workflows/refresh-release-options.yml`, which runs on every tag push and rewrites the options block in `release.yml`.

Why two steps instead of auto-deploy on tag push? Tagging is reversible (delete the tag and re-push); SVN deploys aren't. The manual click is a deliberate guard.

## Build the production zip locally

```sh
npm run build           # → bash scripts/build.sh
# or with a specific version:
bash scripts/build.sh 1.0.4-test
```

Runs the exact same recipe CI uses (`scripts/build.sh` is invoked from both `release.yml` and `ci.yml`). Output: `dist/<slug>-<version>.zip` and a staged tree at `build/staging/<slug>/`. If no version is given, the current `UNIVERSALLY_VERSION` from `universally.php` is used.

Required repo secrets for SVN deploy: `SVN_USERNAME` and `SVN_PASSWORD` (wp.org account credentials).

## Translations

POT regeneration is normally done in CI during the build. To regenerate locally (Docker required):

```sh
cd plugin
npm run makepot
```

## Bundled third-party libraries

The compiled JS in `plugin/panel/build/index.js` bundles:

- [@timkit/conditions](https://www.npmjs.com/package/@timkit/conditions) (Apache-2.0) — condition evaluator for dynamic field visibility.

All other JS dependencies (React, `@wordpress/*`) are externalized at build time and resolved to the script handles WordPress already provides.
