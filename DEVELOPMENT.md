# Development

This document covers building, testing, and releasing the Universally WordPress plugin. For end-user info, see [README.md](README.md).

## Layout

```
plugin/                  # plugin source (ships to wp.org)
  universally.php        # plugin bootstrap
  app/                   # PHP classes (PSR-4, `Universally\` namespace)
  includes/              # procedural helpers, REST entry, conflict handler
  panel/js/              # TypeScript/React sources for the admin panel
  panel/build/           # compiled JS/CSS (generated by `npm run build`)
  panel/src/             # PHP-side panel framework
  assets/                # static plugin assets (icons, switcher JS, etc.)
  languages/             # translations and POT file
  readme.txt             # wp.org listing
assets/                  # wp.org plugin page assets (banners, icons, screenshots)
scripts/release.sh       # release entry тАФ bumps, builds, tags, pushes
docker-compose.yml       # local WordPress for quick testing
```

## Local build

Node.js 18+ and PHP 7.4+ are required.

```sh
cd plugin
npm install
npm run build       # production build
npm run start       # watch mode
composer install    # PHP autoloader
```

`npm run build` regenerates everything inside `plugin/panel/build/` (compiled JS, CSS, source maps, and `*.asset.php` dependency files) from the sources in `plugin/panel/js/` using [@wordpress/scripts](https://www.npmjs.com/package/@wordpress/scripts).

## Test in a real WordPress

```sh
docker compose up -d
open http://localhost:8765
```

Run the installer, then activate the plugin from `Plugins тЖТ Installed Plugins`. The plugin directory is bind-mounted live, so PHP changes take effect immediately; TS/SCSS changes require `npm run build` (or `npm run start` for watch mode).

```sh
docker compose down -v   # wipe DB + WP install when done
```

## Releasing

```sh
npm run release
```

Two-step interactive menu тАФ first pick the **bump type**, then the **channel**:

**Bump type:**
- `1) patch` тАФ `1.0.0 тЖТ 1.0.1`
- `2) minor` тАФ `1.0.0 тЖТ 1.1.0`
- `3) major` тАФ `1.0.0 тЖТ 2.0.0` (extra confirmation)
- `4) custom` тАФ enter the exact version (skips channel prompt)
- `5) rebuild` тАФ force re-release of the current version (skips channel prompt; replaces the tag)

**Channel** (only shown for patch / minor / major):
- `1) stable` тАФ `vX.Y.Z`
- `2) beta` тАФ `vX.Y.Z-beta.N` (auto-increments N from existing beta tags for that base)
- `3) rc` тАФ `vX.Y.Z-rc.N`

Examples:
- patch + stable from `1.0.1` тЖТ `1.0.2`
- patch + beta from `1.0.1` тЖТ `1.0.2-beta.1`
- minor + rc from `1.0.1` тЖТ `1.1.0-rc.1` (or `.2` if `v1.1.0-rc.1` already exists)

The script bumps `plugin/universally.php`, `plugin/package.json`, and `plugin/readme.txt` (`Stable tag` only for stable releases), commits, creates an annotated `vX.Y.Z` tag, and pushes. It does NOT build тАФ `scripts/build.sh` is the single source of truth for that, called by CI on the tag push.

## Build the production zip locally

```sh
npm run build           # тЖТ bash scripts/build.sh
# or with a specific version:
bash scripts/build.sh 1.0.4-test
```

Runs the exact same recipe CI uses (`scripts/build.sh` is invoked from both `release.yml` and `ci.yml`). Output: `dist/<slug>-<version>.zip` and a staged tree at `build/staging/<slug>/`. If no version is given, the current `UNIVERSALLY_VERSION` from `universally.php` is used.

CI (`.github/workflows/release.yml`) picks up the tag push:

- **Stable tag** (`vX.Y.Z`) тЖТ builds, deploys to wp.org SVN trunk + `tags/X.Y.Z` + `assets/` via [10up/action-wordpress-plugin-deploy](https://github.com/10up/action-wordpress-plugin-deploy), creates a GitHub Release with the zip + a one-click WordPress Playground link.
- **Pre-release tag** (`vX.Y.Z-beta.N` / `-rc.N`) тЖТ builds, creates a GitHub Release (marked pre-release) with the zip + Playground link. Skips SVN.

Required repo secrets: `SVN_USERNAME` and `SVN_PASSWORD` (wp.org account credentials).

### Manual prerelease (no local commit)

To produce a downloadable test zip from a feature branch without adding a "Release тАж" commit to its history, dispatch the **Release** workflow from the Actions tab:

- Actions тЖТ **Release** тЖТ **Run workflow** тЖТ pick the branch тЖТ enter a version like `1.0.3-beta.1` (must include a `-` suffix; stable cuts are rejected on dispatch).

CI tags the workflow's commit as `vX.Y.Z-...`, builds, and publishes a GitHub Release (marked pre-release) with the zip + Playground link. Skips SVN.

## Translations

POT regeneration is normally done in CI during the build. To regenerate locally (Docker required):

```sh
cd plugin
npm run makepot
```

## Bundled third-party libraries

The compiled JS in `plugin/panel/build/index.js` bundles:

- [@timkit/conditions](https://www.npmjs.com/package/@timkit/conditions) (Apache-2.0) тАФ condition evaluator for dynamic field visibility.

All other JS dependencies (React, `@wordpress/*`) are externalized at build time and resolved to the script handles WordPress already provides.
